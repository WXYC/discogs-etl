# Migrations

`schema/*.sql` remains the canonical, hand-maintained schema. On top of that we keep an [alembic](https://alembic.sqlalchemy.org/) revision history at `alembic/versions/` so future schema changes have a recorded order, and so that downstream services (Backend-Service, library-metadata-lookup) can rely on a stamped version when introspecting the cache.

Layout:

- `alembic.ini` -- alembic config; the `sqlalchemy.url` placeholder is overridden by `alembic/env.py`.
- `alembic/env.py` -- resolves the URL from `DATABASE_URL_DISCOGS` (canonical) or `DATABASE_URL` (deprecated, warns to stderr); rewrites `postgresql://` to `postgresql+psycopg://` so SQLAlchemy uses psycopg3 instead of pulling in psycopg2.
- `alembic/versions/0001_initial.py` -- baseline migration. Hand-written; its `upgrade()` opens an autocommit psycopg connection and replays `schema/create_functions.sql`, `schema/create_database.sql`, `schema/create_indexes.sql`, `schema/create_track_indexes.sql` in pipeline order. CONCURRENTLY is stripped because the baseline only ever runs against an empty database (mirrors the `strip_concurrently=True` path in `scripts/run_pipeline.py`).
- `alembic/versions/0004_wxyc_identity_match_fns.py` -- deploys the `wxyc_identity_match_{artist,title,with_punctuation,with_disambiguator_strip}` plpgsql function family per wiki §3.3.5. Vendors canonical artifacts from WXYC/wxyc-etl@v0.4.0 (`data/`) under `vendor/wxyc-etl/`; SHA-pinned in `wxyc-etl-pin.txt`. The migration reads the canonical SQL at apply time and short-circuits in alembic offline mode (same pattern as `0003_wxyc_library_v2`). Unaccent is provisioned as a pure-SQL function `wxyc_unaccent_text(text)` (codegen at `lib/unaccent_codegen.py`) rather than a `$SHAREDIR/tsearch_data/`-backed text-search dictionary — the old dict path failed on Railway-managed Postgres because `$SHAREDIR` is root-owned and unwritable even with `pg_write_server_files` (see [#223](https://github.com/WXYC/discogs-etl/issues/223)). The codegen reads `vendor/wxyc-etl/wxyc_unaccent.rules`, emits a `translate()` + small `REPLACE` chain, and patches the vendored SQL's one `unaccent('wxyc_unaccent', r)` call site to `wxyc_unaccent_text(r)` at apply time (the on-disk vendored file stays SHA-pinned). The two-pass equivalence invariant (no overlap between multi-char-dst output chars and single-char-src set) is asserted in the codegen module and gated by `tests/unit/test_wxyc_unaccent_function_codegen.py` — re-vendoring `wxyc_unaccent.rules` must keep that invariant. Index + cache-load flips from `lower(f_unaccent(col))` to `wxyc_identity_match_artist(col)` ship in a follow-on PR (the LML#280 deferred-swap doesn't need them to flip — just the functions). Parity test at `tests/integration/test_wxyc_identity_match_parity.py` covers the 252-row fixture from wxyc-etl plus pin freshness + idempotence.
- `alembic/versions/0005_release_track_artist_role.py` -- adds `extra integer DEFAULT 0` + `role text` columns to `release_track_artist` per [#218](https://github.com/WXYC/discogs-etl/issues/218) so downstream consumers (notably library-metadata-lookup) can filter to main-artist credits (`WHERE extra = 0`) and inspect the source-side `<role>` text on extra credits. Additive: pre-#55 converter CSVs continue to import (the loader treats `extra` / `role` as optional CSV columns) and pre-migration rows default to `extra=0` / `role=NULL`. Re-ETL of the three deployments is required to populate the new columns against existing rows; the LML consumer change in `discogs/cache_service.py:validate_track_on_release` is a separate follow-up.
- `alembic/versions/0006_lookup_negative.py` -- creates `lookup_negative (key_hash bytea PK, artist text, track text, artist_as_keyword boolean, attempted_at timestamptz NOT NULL DEFAULT now(), ttl_seconds integer NOT NULL DEFAULT 604800)` + `idx_lookup_negative_attempted_at` per [WXYC/library-metadata-lookup#341](https://github.com/WXYC/library-metadata-lookup/issues/341) (the A4 ticket). Persists "we asked Discogs and got nothing" verdicts across LML process restarts so Railway redeploys don't cost a full 16-s cascade per cold-cache rare-artist query. Standalone — no FK to any other table; simple `op.execute` (not a side-channel like 0001/0005) so alembic's `--sql` machinery covers it. Companion consumer change lives in LML's `discogs/cache_service.py` (separate PR). Test at `tests/integration/test_alembic_lookup_negative.py` historically stamps at 0005 to skip 0004's text-search dictionary build; post-#223 that workaround is no longer required and could be simplified to run the full chain (deferred — not bundled into the #223 PR).
- `alembic/versions/0007_wxyc_postgres_image_gate.py` -- creates the `wxyc_unaccent` text-search dictionary from `$SHAREDIR/tsearch_data/wxyc_unaccent.rules`, with a plpgsql wrapper that catches `SQLSTATE F0000` (`config_file_error`) and re-raises with an actionable runbook URL pointing at the [wxyc-postgres image runbook](https://github.com/WXYC/wxyc-etl/blob/main/docs/wxyc-postgres-image.md). The dict is **deployment-state**: it requires the destination PG to be running `ghcr.io/wxyc/wxyc-postgres:pg{16,17}` (built by [WXYC/wxyc-etl#127](https://github.com/WXYC/wxyc-etl/issues/127)). Defense-in-depth alongside 0004 — the runtime path still goes through `wxyc_unaccent_text(text)` (function-based, image-independent), but the dict's existence proves the image swap has happened and enables future migrations to use the canonical `unaccent('wxyc_unaccent', ...)` directly. Idempotent (DROP IF EXISTS + DO block); test at `tests/integration/test_alembic_0007_image_gate.py` pins the F0000 SQLSTATE contract, the runbook URL, and the idempotence path.
- `alembic/versions/0008_release_artwork_checked_at.py` -- adds `release.artwork_checked_at timestamptz` (nullable, no default) + the partial index `release_artwork_null_idx ON release (id) WHERE artwork_url IS NULL AND artwork_checked_at IS NULL` per [#239](https://github.com/WXYC/discogs-etl/issues/239). Resolves the bulk-loader vs LML cache-fallthrough ambiguity (NULL `artwork_url` could mean "never asked" or "asked, no image"). The companion LML predicate change ([WXYC/library-metadata-lookup#423](https://github.com/WXYC/library-metadata-lookup/issues/423)) honors the column and stamps `now()` on every API-derived `write_release`; the partial index covers [WXYC/library-metadata-lookup#221](https://github.com/WXYC/library-metadata-lookup/issues/221)'s top-up drain. Additive + idempotent (`ADD COLUMN IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS`); column also dual-written in `schema/create_database.sql` per the dual-write convention. Survives the dedup + prune-copy-swap paths via additions to `scripts/dedup_releases.py:DEDUP_TABLES`, `scripts/verify_cache.py:PRUNE_COPY_TABLES`, and `scripts/verify_cache.py:COPY_TABLE_SPEC` — regression pins at `tests/integration/test_dedup.py::TestDedupCopySwapPreservesMasterId::test_artwork_checked_at_column_persists_after_copy_swap` and `tests/integration/test_verify_cache_columns.py::TestPruneCopyTablesCoversSchema::test_release_keeps_artwork_checked_at`. Migration test at `tests/integration/test_alembic_0008_artwork_checked_at.py`. Re-ETL is not required to populate the column — the bulk loader continues to leave it NULL (the "never asked" semantic); LML's live-API path back-fills on first lookup.
- `alembic/versions/0009_cache_metadata_unique.py` -- fixes a Railway-side drift between `schema/create_database.sql` (which declares `cache_metadata.release_id integer PRIMARY KEY`) and the existing prod table (which predated that declaration and lacked the PK). The 2026-06-07 [#267](https://github.com/WXYC/discogs-etl/issues/267) manual-jumpstart run 3 surfaced this when `scripts/import_csv.py:populate_cache_metadata` hit `InvalidColumnReference: there is no unique or exclusion constraint matching the ON CONFLICT specification` — the populate path's `INSERT ... ON CONFLICT (release_id) DO NOTHING` (added per [#188](https://github.com/WXYC/discogs-etl/issues/188) to tolerate concurrent `api_fetch` writes from LML's runtime) needs a UNIQUE or PRIMARY KEY on `release_id`. Two-step idempotent: (1) `DELETE FROM cache_metadata a USING cache_metadata b WHERE a.ctid < b.ctid AND a.release_id = b.release_id AND (a.source != 'api_fetch' OR b.source = 'api_fetch')` to dedup, preserving `api_fetch` rows (which LML caches across rebuilds — `populate_cache_metadata` docstring at `import_csv.py:506-533`); (2) `DO $$ ... IF NOT EXISTS ... ALTER TABLE cache_metadata ADD CONSTRAINT cache_metadata_release_id_unique UNIQUE (release_id)` to add the missing constraint without disturbing fresh DBs that already have the PK. Side-channel autocommit psycopg, refuses offline mode. The DELETE is row mutation of persisted data, so this revision does NOT meet the "pure DDL" gate in [`docs/migrations-runbook.md`](migrations-runbook.md)'s out-of-band procedure — it has to ride the rebuild. Migration test at `tests/integration/test_alembic_0009_cache_metadata_unique.py` pins the dedup logic + UNIQUE-add + dual-write parity (the test pre-applies `schema/create_database.sql` before stamping to 0008 + upgrading, so the alembic chain and the legacy path can't drift on the constraint). Revision id intentionally short (`0009_cache_metadata_unique`) because alembic_version is `varchar(32)` and the original revision id overflowed (fixed in [c228b47](https://github.com/WXYC/discogs-etl/commit/c228b47)).
- `alembic/versions/0010_release_not_found.py` -- adds `release.not_found boolean NOT NULL DEFAULT FALSE` for [WXYC/library-metadata-lookup#510](https://github.com/WXYC/library-metadata-lookup/issues/510)'s 404 tombstone discriminator. Distinguishes "never asked" (no row in `release`) from "asked, Discogs returned 404" (row present with `not_found = TRUE`) so LML's negative-cache stays warm and subsequent callers don't re-burn the rate-limit budget on the same 404. The companion LML write (`cache_service.write_release`) emits a narrow tombstone row with identifier columns intentionally NULL/empty so a future 200-response can fill them in without losing them mid-flight; the rebuild-side UPSERT path in `scripts/import_csv.py:import_release_via_upsert` writes `not_found = FALSE` into the SET list so a fresh dump implicitly clears any prior tombstone. Side-channel autocommit psycopg with a `BEGIN ... INSERT INTO release (id, title, not_found) VALUES (-2147483648, '', TRUE) ... ROLLBACK` precondition probe — surfaces a future-added `CHECK (title <> '')` at migration time rather than on the first prod 404 write. Default `FALSE` makes the migration safe to ship before LML is updated. Column is dual-written in `schema/create_database.sql` (line 66, with a comment pointing at this migration). Migration test at `tests/integration/test_alembic_0010_release_not_found.py` covers the column shape, the tombstone round-trip with empty title, and idempotence-against-dual-written-schema (pre-applies `create_database.sql` before stamping to 0009 + upgrading).
- `alembic/versions/0011_artist_not_found.py` -- parallel to 0010 but for `artist.not_found boolean NOT NULL DEFAULT FALSE`, same LML#510 motivation. The artist side has two discriminators that compose: `fetched_at` (LML#503) distinguishes ETL stubs from rows hydrated by LML's live-API path; `not_found` (this migration) distinguishes 200-response hydration from 404 tombstone hydration. LML's tombstone-write UPSERT preserves a hydrated parent's `name` / `profile` / `image_url` by omitting them from the `ON CONFLICT DO UPDATE SET` clause; the non-tombstone branch adds `not_found = FALSE` so a recovered 200 clears any prior tombstone in one statement. The rebuild-side stub-insert in `scripts/import_csv.py:import_artist_details:911-917` clears tombstones via `ON CONFLICT (id) DO UPDATE SET not_found = FALSE WHERE artist.not_found = TRUE` — narrowed to actual tombstones so non-tombstone rows aren't disturbed. Side-channel autocommit psycopg with the same empty-name precondition probe as 0010, plus the empty-name write contract pinned in the migration test. Dual-written in `schema/create_database.sql` (line 157). Migration test at `tests/integration/test_alembic_0011_artist_not_found.py`.
- `alembic/versions/0012_entity_release_identity.py` -- brings the release-side cross-cache identity layer under the alembic chain for [#278](https://github.com/WXYC/discogs-etl/issues/278) / [WXYC/library-metadata-lookup#526](https://github.com/WXYC/library-metadata-lookup/issues/526). Three objects under a new `entity` schema: `entity.release_identity` (PK `id` + six per-source UNIQUE columns — `discogs_release_id`, `discogs_master_id`, `musicbrainz_release_id`, `spotify_album_id`, `apple_music_album_id`, `bandcamp_album_url` — plus `reconciliation_status`, `created_at`, `updated_at`); `entity.release_reconciliation_log` (audit trail with `identity_id INTEGER NOT NULL REFERENCES entity.release_identity(id)`, `NO ACTION` delete rule matching the artist-side convention); and `idx_release_reconciliation_log_identity_id` (FK referencing index — Postgres does not auto-index FK columns, so every `WHERE identity_id = $1` lookup needs it). The six UNIQUE columns are load-bearing for LML's mint protocol (`INSERT ... ON CONFLICT ({col}) DO NOTHING RETURNING id` in `entity/store.py::mint_or_get_release_identity`); dropping any raises a loud 500 (`RuntimeError("UNIQUE constraint appears broken")`) on the LML write surface rather than silently double-minting. DDL is mirrored from LML's canonical `entity/release_identity.sql` ([WXYC/library-metadata-lookup#530](https://github.com/WXYC/library-metadata-lookup/pull/530)) with one addition: `CREATE SCHEMA IF NOT EXISTS entity` precedes the table DDL because the canonical file assumes the schema is already present (existing prod has it from artist-side out-of-band bootstrap; fresh dev DBs do not). All DDL idempotent; downgrade drops index → log → identity in FK order and intentionally leaves the `entity` schema in place because the artist-side tables (`entity.identity` / `entity.reconciliation_log`) still live there. Adopting those into the alembic chain happens in 0013 (see below). `discogs_release_id` / `discogs_master_id` are `INTEGER` but **not** FKs to `release.id` / `master.id` — the identity layer's lifecycle is independent of the monthly cache rebuild, and FKs would couple the two. Side-channel autocommit psycopg via the shared `lib/alembic_helpers.py` (`refuse_offline`, `resolve_db_url`). Dual-written in `schema/create_database.sql` lines 292–334. Migration test at `tests/integration/test_alembic_0012_entity_release_identity.py` covers the schema bootstrap, column shape, all six per-source UNIQUE constraints (asserted against `pg_constraint`), FK `NO ACTION` semantics, FK index presence, an end-to-end mint→remint smoke (`INSERT ... ON CONFLICT (discogs_release_id) DO NOTHING RETURNING id` minting first call and returning empty on the repeat), and re-application idempotence against a populated entity schema.
- `alembic/versions/0013_adopt_entity_identity.py` -- closes the artist-side counterpart of 0012 for [#279](https://github.com/WXYC/discogs-etl/issues/279). `entity.identity` and `entity.reconciliation_log` already exist on prod via out-of-band bootstrap (no prior revision in this repo created them); LML treats them read-only (`identity/dependencies.py` issues a `SELECT 1 FROM entity.identity LIMIT 0` probe that flips the routes to 503 when missing — LML never bootstraps). This revision adopts them under the alembic chain so a future cold rebuild picks them up automatically rather than needing a manual `psql` step, the canonical DDL lives in the repo, and future tweaks can live in subsequent revisions instead of out-of-band patches. DDL is mirrored from `wxyc-etl/src/schema/entity.rs::ENTITY_IDENTITY_DDL` + `RECONCILIATION_LOG_DDL` (the canonical mirror of the prod artist-side shape) embedded by copy — not import — so the revision survives wxyc-etl bumps without silently changing. Two indexes added: `idx_entity_identity_status` on `(reconciliation_status)` for LML's reconciler dashboard scans, and `idx_entity_reconciliation_log_identity_id` covering the FK referencing column. Both `IF NOT EXISTS` so the migration is a clean no-op against existing prod tables (the adoption case) AND lands the schema from scratch on fresh dev DBs. Downgrade is intentionally a **no-op** — adoption migrations don't drop the tables they adopt: dropping the indexes alone would re-introduce a perf cliff against tables that physically remain, and `DROP INDEX IF EXISTS` cannot distinguish indexes this revision created from inherited ones (would punish prod for adopting tables that already had the indexes from out-of-band bootstrap). Side-channel autocommit psycopg via the shared `lib/alembic_helpers.py`. Dual-written in `schema/create_database.sql` lines 339–369. Migration test at `tests/integration/test_alembic_0013_adopt_entity_identity.py` covers the fresh-dev path (column shape, `library_name UNIQUE`, FK `NO ACTION` rules, both indexes) AND the prod-adoption path (pre-seeded tables with a sentinel row; upgrade must no-op against existing data and idempotently land the indexes), plus the no-op downgrade contract (tables, indexes, and rows all survive `alembic downgrade 0012_entity_release_identity`).

Apply against an empty Postgres:

```bash
createdb discogs_cache_migrations_test
DATABASE_URL_DISCOGS=postgresql://localhost:5433/discogs_cache_migrations_test \
  alembic upgrade head
```

Add a new migration:

```bash
alembic revision -m "<short-name>"
# edit alembic/versions/<rev>_<short-name>.py and write upgrade()/downgrade()
```

Prefer hand-written `op.execute()` migrations -- `--autogenerate` is intentionally off (no SQLAlchemy ORM models exist in this repo). When the change is simple SQL, it's fine to embed the DDL directly in `upgrade()`. When it's a large refactor, drop a new file under `schema/` and have `upgrade()` `op.execute(open(...).read())` it, the same pattern as `0001_initial`.

**Deploy wiring**: the monthly rebuild workflow (`.github/workflows/rebuild-cache.yml`) runs `alembic upgrade head` before the pipeline kicks off, gated by a `Verify alembic baseline is stamped` step that aborts the workflow within ~30 seconds (before the multi-GB dump download) if the target DB lacks an `alembic_version` table. Once stamped, the upgrade applies any new migrations added since the last rebuild. The pipeline itself still applies `schema/*.sql` directly via `run_sql_file` for fresh-rebuild DDL — alembic and the legacy path stay in parity by way of the dual-write convention below.

**Dual-write convention**: when adding a schema change, write the new `alembic/versions/<rev>_*.py` (or its referenced `schema/*.sql` snippet) AND mirror the change into `schema/create_database.sql` / `create_indexes.sql` / etc. so a fresh rebuild produces the same end-state as the alembic upgrade chain.

**One-shot stamp procedure** (one-time operator action, see `docs/migrations-runbook.md`): existing production `discogs-cache` databases must be `alembic stamp head`-ed once, with a backup snapshot taken first, before the rebuild workflow runs `alembic upgrade head` for the first time. Without the stamp the rebuild workflow's pre-flight `Verify alembic baseline is stamped` step aborts; the migration itself is also defensive (see below) but the explicit operator stamp keeps the audit trail clean.

**Defensive guards in `0001_initial.py`**: because `upgrade()` applies `schema/*.sql` via a side-channel `psycopg.connect(..., autocommit=True)` rather than `op.execute(...)`, the standard alembic `--sql` (offline mode) flag cannot intercept it. To prevent the 2026-04-28 prod-cache wipe scenario, `upgrade()` raises `RuntimeError` if `context.is_offline_mode()` is true, and short-circuits if the discogs-cache schema (`release` + `cache_metadata`) is already present. Future migrations that need autocommit DDL (e.g. `CREATE INDEX CONCURRENTLY`, extension creation) should follow the same `is_offline_mode()` check, or write the migration with `op.execute(..., execution_options={"isolation_level": "AUTOCOMMIT"})` so alembic's offline mode can intercept it. The two pinning tests live in `tests/integration/test_alembic_baseline.py`.

**Shared helpers (`lib/alembic_helpers.py`)**: 0001 through 0009 each carry an inline copy of the same two-helper pair — `_refuse_offline(direction)` (raises if `context.is_offline_mode()` is true) and `_resolve_db_url()` (reads `DATABASE_URL_DISCOGS` / `DATABASE_URL`, raises if absent) — except 0001 (which inlines the logic directly into `upgrade()`) and 0006 (which uses `op.execute` and doesn't need either). 0010, 0011, and 0012 are the first revisions to share via `lib/alembic_helpers.py` (`refuse_offline(revision_label, direction)`, `resolve_db_url(revision_label)`). The `revision_label` argument flows the migration's revision id into the error message so operators see which migration is missing the env var or trying to run `--sql`. Future side-channel migrations should import from `lib/alembic_helpers.py` rather than re-inlining; the helpers module deliberately doesn't import alembic so it stays callable from unit tests, CLI scripts, and any other non-MigrationContext caller.
